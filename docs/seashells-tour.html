Microsite Tour + Brand Direction

The site already knows what it is.

The current pb microsite is warm, specific, and practical. It sells one tiny motion really well: stop selecting things your agent already knows how to hand you.

The next move is not to throw that away. The next move is to explain that pb, look, history cleanup, and shell hygiene are all part of the same shoreline: shell-adjacent artifact handling. Publicly that becomes Seashells. Operationally the skill stays shellskills. Verb-wise, pb stays perfect for copy because it means past buffer.

What this document does

It walks the current homepage section by section, then sketches the clearer brand architecture sitting just beneath it.

Current feeling Butter knife. Toast stripe. Golden utility. One tiny helper with a point of view.
Brand gap pb and agent-look behave like siblings, but the family name is still implicit.
Proposed answer Seashells as the public family, shellskills as the stable skill root, pb and look as capabilities.

Brand Direction

What the site wants to become

The present brand is flavorful but narrow. pb names one capability. The broader concept is stronger: tools that manage the membrane between your shell, your artifacts, and your agent.

That family should be named once at the top. Everything else becomes a capability under it.

Public family name: Seashells

The public brand should sound tactile, memorable, and slightly playful. Seashells keeps the hand-feel of the butter metaphor, but broadens it from one tool to a shoreline of shell helpers.

Stable skill root: shellskills

This is the internal and practical name. It is literal, composable, and future-safe. It can absorb new behaviors without renaming the top-level skill every time another useful shell affordance appears.

Capability name: pb

Keep it. The interpretation as past buffer is clean, shell-native, and short. It sounds like a command because it should.

Capability name: look

If screenshots are a clipboard function, then look belongs in the same family. It becomes the intake side of the same artifact loop that pb handles on output.

Capability names: history, session, env

These belong naturally once the promise expands from clipboard convenience to shell workflow management. They are not side quests. They are the safety rails around the same work surface.

Seashells = family brand shellskills = master skill pb = copy verb look = screenshot intake history = guardrails
Core pitch: Seashells is a family of shell-adjacent agent tools that move artifacts cleanly between conversation, clipboard, screenshots, history, and your zsh environment.
Microsite Architecture

How the site could evolve without losing its current charm

The homepage does not need to become a generic product suite grid overnight. It can keep the current tactile tone, but reorganize around one family heading and a small number of shell verbs.

Front Door
Seashells homepage A family landing page with a clear thesis: your agent should manage shell-adjacent artifacts, not make you drag them around.
Primary Verb
pb / past buffer The current star. Keep the butter-knife tone as one card or lane inside the larger family.
Input Verb
look Position screenshots as the intake complement to clipboard output. Same promise, opposite direction.
Shell Care
History, session, env Show that the family also protects the shell itself: history safety, session cleanup, and env hygiene.
Visual Language

What I would preserve and what I would shift

Keep: the tactile metaphors, the specific verbs, the no-nonsense install snippets, and the sense that these are real tools for real shell work.

Shift: from all-butter-all-the-time to a shoreline palette. The current gold should not disappear; it becomes one accent among shell pink, sea green, and deep teal. That lets pb remain warm while making room for look and shell safety features to feel like part of the same family.

Typography should also carry more of the identity. A serif display face for the family voice and a modern grotesk for the command surfaces gives the site a more editorial, less "single landing page template" feel.

Naming Rules

The naming system should stay boring under the hood

Public names can be lyrical. Operational names should stay plain.

Publicly

Seashells is the family. Marketing pages, social posts, overview docs, and ecosystem language can use it freely.

In skills and prompts

shellskills remains the stable top-level invocation surface. This avoids constant rename churn in installed local tooling.

In capability verbs

pb, look, history, env, and session stay terse because the shell is where terseness pays off.

Suggested Rollout

A migration path that does not waste the current site

Phase 1

Publish overview pages like this one. Introduce Seashells as the family name without forcing an immediate repo rename.

Phase 2

Add a family-level microsite or top banner that frames pb and agent-look as capabilities, not isolated toys.

Phase 3

Move the install and docs language toward shellskills as the stable skill root, while preserving pb as the copy verb and public shorthand.

The site does not need a rebrand because the old one failed. It needs one because the product family got bigger than the first noun that named it.
